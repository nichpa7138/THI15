Session 1

📖 ทฤษฎีวงจรการพัฒนาระบบ (SDLC 7 ขั้นตอน) ในงานสาธารณสุข

วงจรการพัฒนาระบบ SDLC (Systems Development Life Cycle) อ้างอิงตามมาตรฐานของ IBM คือกระบวนการที่เป็นระบบในการวางแผน สร้าง ทดสอบ และปรับใช้ระบบสารสนเทศ โดยในบริบทของ งานสาธารณสุข สุขภาพ และเวชระเบียนดิจิทัล มีรายละเอียด 7 ขั้นตอนหลักดังนี้:

ขั้นตอน (Phase) กิจกรรมหลักในงานสุขภาพดิจิทัล (Key Activities in Health IT) ผลลัพธ์ที่ได้ (Deliverables)
1. Planning (วางแผน) กำหนดขอบเขตระบบระเบียนสุขภาพ ค้นหาความเสี่ยงในการจัดการข้อมูลผู้ป่วย และวิเคราะห์ความปลอดภัย PDPA แผนงานเริ่มต้น / เอกสาร SRS
2. Analysis (วิเคราะห์) ศึกษาเกณฑ์คัดกรองทางการแพทย์มาตรฐาน (เช่น THI-15) พฤติกรรมผู้ใช้ และความต้องการของแพทย์/พยาบาล เอกสารข้อกำหนดความต้องการที่สมบูรณ์
3. Design (ออกแบบ) ออกแบบหน้าจอกรอกประวัติ (EHR UI) กำหนดความสัมพันธ์ของตารางข้อมูล แยกข้อมูลทั่วไปออกจากผลการประเมินเพื่อความปลอดภัย เอกสารการออกแบบ (SDD) / Prototype
4. Coding (เขียนโค้ด) เขียนโค้ดจริงโดยประยุกต์ใช้เครื่องมือ AI ช่วยแปลงภาษามนุษย์เป็นสคริปต์คำนวณและบันทึกคะแนนกลับกลุ่มข้อลบ-บวกลง Google Sheets ระบบต้นแบบเว็บบริการสุขภาพจริง
5. Testing (ทดสอบ) ทดสอบความแม่นยำของสูตรแปรผลคะแนนทางระบาดวิทยา/สุขภาพจิต ทำลายบั๊ก และอุดช่องโหว่การรั่วไหลของข้อมูลคนไข้ แอปพลิเคชันที่เสถียรและปลอดภัย
6. Deployment (ติดตั้ง) นำเว็บแอปพลิเคชันระบบสุขภาพขึ้นคลาวด์เปิดให้กลุ่มนักศึกษาและบุคลากรทดลองใช้ พร้อมฝึกอบรมการบันทึกเวชระเบียนที่ถูกต้อง ระบบที่พร้อมให้บริการแก่สาธารณะ (GA)
7. Maintenance (บำรุงรักษา) ติดตามผลและอัปเดตระบบอย่างต่อเนื่อง เช่น ปรับสเกลช่วงคะแนนตามคู่มือแพทย์ใหม่ๆ และสำรองข้อมูลสุขภาพอย่างเป็นระบบ ระบบที่มีเสถียรภาพและทันสมัยเสมอ

1. Planning (การวางแผนโครงการระบบระเบียนสุขภาพ)

เป็นการกำหนดเป้าหมายเพื่อตอบโจทย์ปัญหาทางสาธารณสุข เช่น ปัญหาความเครียดสะสมในวัยเรียน โดยกำหนดขอบเขตว่าแอปพลิเคชันจะต้องจัดเก็บข้อมูลสารสนเทศอะไรบ้าง เพื่อป้องกันไม่ให้ขอบเขตโครงการบานปลาย โดยผู้มีส่วนได้ส่วนเสียประกอบด้วย นักวิเคราะห์ระบบเวชระเบียน แพทย์ และอาจารย์ผู้ดูแลหลักสูตร

2. Analysis (การวิเคราะห์ข้อกำหนดและความเป็นไปได้ทางคลินิก)

การเปลี่ยนแนวคิดเป็นแบบแปลนที่พร้อมทำจริง โดยรวบรวมเกณฑ์แบบประเมินสุขภาพจิตมาตรฐานจากกรมสุขภาพจิตและข้อกำหนดความปลอดภัยทางกฎหมายสารสนเทศสุขภาพ (เช่น ความลับผู้ป่วย) ผลลัพธ์สุดท้ายคือเอกสารข้อกำหนดความต้องการทางเทคนิคและเวิร์กโฟลว์การไหลของคนไข้ (Patient Workflow) ที่สมบูรณ์

3. Design (การออกแบบส่วนเชื่อมต่อและฐานข้อมูลระเบียนสุขภาพ)

การออกแบบโครงสร้างเชิงเทคนิค เช่น การนำทางในเว็บ (Navigation) ตารางความสัมพันธ์ของฐานข้อมูล และการจำลองสถานการณ์คุกคามทางไซเบอร์ (Threat Modeling) เช่น การเข้ารหัสข้อมูลรหัสนักศึกษาและวันเกิด เพื่อจำกัดสิทธิ์การเข้าถึงประวัติสุขภาพเฉพาะรายบุคคลอย่างแน่นหนา

4. Coding (การเขียนโค้ดพัฒนาระบบสุขภาพดิจิทัล)

การลงมือพัฒนาจริงโดยอ้างอิงจากแผนภาพการออกแบบ เลือกใช้เทคโนโลยีที่ยืดหยุ่น เช่น Google Apps Script และ HTML + Tailwind CSS รวมถึงการใช้เทคนิค Generative AI ช่วยร่างลอจิกซับซ้อน เช่น การแปลผลและวิเคราะห์เกณฑ์ย้อนกลับอัตโนมัติ ช่วยย่นระยะเวลาการพิมพ์โค้ด (Vibe Coding) ให้รวดเร็วยิ่งขึ้น

5. Testing (การทดสอบคุณภาพและความสอดคล้องทางการแพทย์)

ขั้นตอนที่ทีมควบคุมคุณภาพ (QA) และทีมนักวิชาการเวชระเบียน จะร่วมกันตรวจสอบความถูกต้องของระบบ โดยเน้นย้ำ **ความถูกต้องของสูตรคำนวณคะแนนประเมิน (Medical Formula Validation)** เป็นหลัก เพราะหากแปรผลผิดพลาดจะส่งผลกระทบต่อสถิติดัชนีทางสุขภาพ และความน่าเชื่อถือทางระบาดวิทยาอย่างร้ายแรง

6. Deployment (การติดตั้งและประเมินผลการใช้งานในพื้นที่จริง)

การนำรหัสโปรแกรมที่ผ่านการทดสอบขึ้นสู่คลาวด์เซิร์ฟเวอร์ และทดสอบปล่อยระบบในรูปแบบนำร่อง (Beta Test) เพื่อประเมินความสอดคล้องในการใช้งานจริงบนอุปกรณ์มือถือและอุปกรณ์ข้างเคียง ก่อนเปิดตัวใช้งานอย่างเป็นทางการ (General Availability - GA) เพื่ออำนวยความสะดวกให้แก่นักศึกษากลุ่มเป้าหมายทั้งหมด

7. Maintenance (การบำรุงรักษาระบบและสำรองฐานข้อมูลอย่างต่อเนื่อง)

ระบบเวชระเบียนสุขภาพจำเป็นต้องมีการตรวจสอบดูแลอยู่ตลอดเวลาเพื่อรับประกันการเข้าถึงข้อมูลที่ลื่นไหล การออกสติกเกอร์แพตช์อุดรอยรั่วความปลอดภัย การพัฒนาปรับปรุงระบบตามเกณฑ์สุขภาพจิตใหม่ๆ ที่ปรับเปลี่ยนในอนาคต และสร้างระบบ Continuous Integration (CI/CD) เพื่อสำรองข้อมูลให้มีประสิทธิภาพและพร้อมกู้คืนได้เสมอ

📊 กรณีศึกษา: โมเดล IPO ของแบบประเมินสุขภาพจิต THI-15

การวิเคราะห์ระบบที่ดีเริ่มจากการเข้าใจโมเดล IPO (Input-Process-Output-Storage) ซึ่งจะจำแนกเส้นทางการรับและแสดงผลข้อมูลของแอปพลิเคชันอย่างเป็นตรรกะ ดังรายละเอียดต่อไปนี้:

📥 Input (นำเข้า)
• ลำดับประเมิน (no)
• วันที่และเวลา (date_time)
• รหัสนักศึกษา (st_id)
• ชื่อนักศึกษา (name)
• นามสกุล (lname)
• วันเกิด (dob)
• อีเมล (email)
• หลักสูตร (course)
• คำตอบประเมิน 15 ข้อ (q1 - q15)
⚙️ Process (ประมวลผล)
ขั้นตอนที่ 1: การให้คะแนนรายข้อ

ข้อคำถามเชิงบวก (12 ข้อ): q1, q4, q5, q6, q7, q8, q9, q10, q11, q13, q14, q15
👉 ไม่เลย = 1 | เล็กน้อย = 2 | มาก = 3 | มากที่สุด = 4

ข้อคำถามเชิงลบ (3 ข้อกลับ): q2, q3, q12
👉 ไม่เลย = 4 | เล็กน้อย = 3 | มาก = 2 | มากที่สุด = 1

ขั้นตอนที่ 2: แปลผลคะแนนรวม (เต็ม 60)

• 51 – 60 คะแนน: มีความสุขมากกว่าคนทั่วไป (Good)
• 44 – 50 คะแนน: มีความสุขเท่ากับคนทั่วไป (Fair)
• 43 คะแนน หรือต่ำกว่า: มีความสุขต่ำกว่าคนทั่วไป (Poor)

📤 Output & Storage
Output (แสดงผลที่หน้าจอ) • คะแนนรวมสะสมที่คำนวณได้ (total)
• ผลประเมินแปลผลความสุข (result)
💾 Storage (แหล่งเก็บรักษา)

เมื่อเกิดการทำงานและคำนวณผลลัพธ์แล้ว ข้อมูลจะส่งข้ามระบบและบันทึกลงสู่ตารางฐานข้อมูลใน Google Sheets ทันที เพื่อป้องกันข้อมูลสูญหายและเปิดดูย้อนหลังได้

👥 กิจกรรมปฏิบัติงานคู่ (Assignment 1):

ให้นักศึกษาทำการจับคู่กัน จากนั้นระดมสมองและเลือกค้นหาแบบสอบถามสุขภาพ สติปัญญา หรือระดับความซึมเศร้าที่เป็นแบบประเมินมีมาตรฐาน แล้วนำข้อมูลชุดนั้นมาวิเคราะห์และเขียนแยกองค์ประกอบตาม **IPO Model (Input, Process, Output, Storage)** โดยระบุด้วยว่าระบบของตนเองจำเป็นต้องสร้างตารางเก็บสิ่งใดเพื่อไม่ให้ข้อมูลหาย

Session 2

📋 มาตรฐานการออกแบบตารางและกำหนดฟิลด์ข้อมูล

💡 กฎเหล็กการกำหนดฟิลด์ข้อมูลตามมาตรฐานสากล:

  • ใช้ภาษาอังกฤษตัวพิมพ์เล็กเป็นมาตรฐานสากล: หลีกเลี่ยงตัวพิมพ์ใหญ่สลับเล็กเพื่อลดข้อผิดพลาด และ ห้ามใช้ภาษาไทยเด็ดขาด เช่น ใช้ st_id แทนที่จะเขียน รหัสนักศึกษา หรือ StudentID
  • ห้ามเว้นวรรค ให้เชื่อมด้วยสัญลักษณ์ Underscore (_) เสมอ: เช่น ป้องกันข้อผิดพลาดจากการแปลชุดคำสั่งด้วยการสะกดว่า date_time หรือ id_course (ห้ามพิมพ์แบบมีช่องว่างเด็ดขาด)
  • หลีกเลี่ยงคำต้องห้ามที่เป็น Reserved Word: คำบางคำเป็นฟังก์ชันเฉพาะของโปรแกรมฐานข้อมูล เช่น Date, Time, Table, User ควรกำหนดส่วนเสริมให้เฉพาะเจาะจงขึ้น เช่น date_time หรือ user_status

📊 พจนานุกรมข้อมูลมาตรฐานครบทุกตาราง (Data Dictionary)

รายละเอียดของโครงสร้างฐานข้อมูลทั้ง 3 ตารางเพื่อใช้ในการสร้างคอลัมน์ใน Google Sheets

🗂️ 1. ตารางเก็บหลักสูตร (Table: course)
ชื่อฟิลด์ (English Field) คำอธิบายภาษาไทย ชนิดข้อมูล (Data Type) รายละเอียดและข้อกำหนด (Constraints)
id_course รหัสหลักสูตร Text 🔑 Primary Key (เช่น C001, C002) ห้ามเว้นว่าง ห้ามซ้ำ
course ชื่อหลักสูตร/สาขาวิชา Text ชื่อวุฒิการศึกษาเต็ม (เช่น "วิทยาศาสตรบัณฑิต สาขาวิชาเวชระเบียน")
👥 2. ตารางเก็บข้อมูลนักศึกษา (Table: member)
ชื่อฟิลด์ (English Field) คำอธิบายภาษาไทย ชนิดข้อมูล (Data Type) รายละเอียดและข้อกำหนด (Constraints)
st_id รหัสนักศึกษา Text 🔑 Primary Key (เช่น "5600001") ห้ามเว้นว่าง ห้ามซ้ำ ใช้ในการล็อกอิน
name ชื่อจริงนักศึกษา Text ชื่อผู้ใช้ระบบภาษาไทย ไม่ระบุคำนำหน้าชื่อ
lname นามสกุลนักศึกษา Text นามสกุลภาษาไทย
dob วันเดือนปีเกิด Text รูปแบบตัวเลขล้วน ววดดปปปป (เช่น "29102534") ทำหน้าที่เป็นรหัสผ่าน (Password)
email อีเมลสถาบัน Text อีเมลองค์กร (เช่น nichapa.ja@kmpht.ac.th) เพื่อตรวจสอบความปลอดภัย
id_course รหัสหลักสูตร Text 🔗 Foreign Key เพื่อเชื่อมหาตาราง course สัมพันธ์ข้อมูลกัน
status สถานะผู้ใช้งาน Text กำหนดสิทธิ์ของบัญชี ค่าเริ่มต้นคือ "student"
📊 3. ตารางบันทึกผลการประเมิน (Table: THI-15)
ชื่อฟิลด์ (English Field) คำอธิบายภาษาไทย ชนิดข้อมูล (Data Type) รายละเอียดและข้อกำหนด (Constraints)
no ลำดับรายการ Number / Auto 🔑 Primary Key ลำดับการประเมิน รันเพิ่มอัตโนมัติเมื่อมีการส่งผล
date_time วันที่เวลา DateTime ประทับเวลาที่ทำรายการอัตโนมัติ (เช่น "08/07/2026 12:34:25")
st_id รหัสนักศึกษา Text 🔗 Foreign Key ระบุรหัสนักศึกษาที่เป็นเจ้าของคะแนนประเมินนี้

🔄 2. แผนผัง System Flow: การเดินทางของข้อมูลในระบบ

แผนผังการไหลของข้อมูลจำลองสถานะและการเดินทางของตัวแปรตั้งแต่หน้าจอบราวเซอร์ไปจนถึงคลาวด์ดาต้าเบส:

1
ขั้นตอนล็อกอิน (Login Flow)

กรอก รหัสนักศึกษา (st_id) และวันเกิด (dob) ➡️ หน้าจอส่งคำขอตรวจสอบไปยัง Google Apps Script ➡️ ตรวจค้นหากับข้อมูลในชีท member ➡️ ส่งผลตอบกลับทาง Client

2
ขั้นตอนแสดงผลและตรวจสอบประวัติ (History Loading)

เมื่อยืนยันสิทธิ์ผ่านระบบ ➡️ ระบบเรียกใช้สคริปต์ดึงประวัติเก่าจากชีท THI-15 โดยกรองเฉพาะชุดข้อมูลที่ฟิลด์ st_id ตรงกันกับเซสชันล็อกอิน ➡️ แสดงสถิติและคะแนนประเมินย้อนหลัง

3
ขั้นตอนการตอบแบบประเมินและประมวลผล (Evaluation Submit)

ผู้ใช้ทำแบบประเมินและกดส่งคำตอบ ➡️ ระบบคำนวณสลับข้อคำถามเชิงบวก-ลบที่เครื่องเซิร์ฟเวอร์ ➡️ จัดกลุ่มแปรผลตามเกณฑ์ ➡️ ทำบันทึกข้อมูลและส่งเพิ่มแถวแบบสอบถามไปชีท THI-15 ทันที

🏠 1. สถาปัตยกรรมระบบ Client-Server (อธิบายผ่านร้านอาหาร)

การทำงานประสานของเว็บแอปพลิเคชันที่มีระบบฐานข้อมูลประกอบ สามารถเปรียบเทียบกับส่วนประกอบของ ร้านอาหาร ได้อย่างเป็นสัดส่วน ดังนี้:

🏠 Client (หน้าร้าน / HTML + JS)

เปรียบเสมือน สมุดเมนูที่อยู่บนโต๊ะอาหาร และบริเวณที่ลูกค้านั่งรับชม สื่อสารกับลูกค้า สแกนเลือกส่งข้อมูลผ่านอินเทอร์เฟซ

👨‍🍳 Server (ห้องครัว / Apps Script)

เปรียบเสมือน เชฟผู้ปรุงอาหารอยู่ในครัว รับคำสั่งมาทำคำนวณสูตร ล็อกอิน และจัดการระบบตามเงื่อนไขอย่างถูกต้องเป็นส่วนสัด

🧊 Database (คลังอาหาร / Google Sheets)

เปรียบเสมือน ตู้เย็นเก็บวัตถุดิบ เป็นตู้เก็บข้อมูลทุกอย่างที่สำคัญ มีความเสถียรและปลอดภัยเป็นหลักแหล่งเพื่อดึงไปปรุงและจัดเรียงได้

function getCourses() {
  try {
    const ss = SpreadsheetApp.getActiveSpreadsheet();
    const sheet = ss.getSheetByName('course');
    if (!sheet) return [];
    const data = sheet.getDataRange().getValues();
    const courses = [];
    for (let i = 1; i < data.length; i++) {
      if (data[i][0]) {
        courses.push({
          id: data[i][0].toString().trim(),
          name: data[i][1] ? data[i][1].toString().trim() : ''
        });
      }
    }
    return courses;
  } catch (e) {
    return [];
  }
}

function loginUser(st_id, dob) {
  try {
    const ss = SpreadsheetApp.getActiveSpreadsheet();
    const sheet = ss.getSheetByName('member');
    if (!sheet) return { success: false, message: 'ไม่พบตารางข้อมูลสมาชิก member' };
    const data = sheet.getDataRange().getValues();
    const inputId = String(st_id).trim();
    const inputDob = String(dob).trim();
    for (let i = 1; i < data.length; i++) {
      let rowId = String(data[i][0]).trim();
      let rowDob = String(data[i][3]).trim();
      if (rowId === inputId && rowDob === inputDob) {
        return {
          success: true,
          user: {
            st_id: data[i][0],
            name: (data[i][1] || '') + ' ' + (data[i][2] || ''),
            course: data[i][5] || ''
          }
        };
      }
    }
    return { success: false, message: 'รหัสนักศึกษา หรือ วันเกิดไม่ถูกต้อง' };
  } catch (error) {
    return { success: false, message: 'เกิดข้อผิดพลาดของระบบ: ' + error.message };
  }
}

  }
}

function getSystemSummary() {
  try {
    const ss = SpreadsheetApp.getActiveSpreadsheet();
    const thiSheet = ss.getSheetByName('THI-15');
    const memberSheet = ss.getSheetByName('member');
    const courseSheet = ss.getSheetByName('course');
    if (!thiSheet || !memberSheet || !courseSheet) {
      return { success: false, message: 'ไม่พบชีทที่จำเป็นในการวิเคราะห์ระบบ' };
    }
    const thiData = thiSheet.getDataRange().getValues();
    const memberData = memberSheet.getDataRange().getValues();
    const courseData = courseSheet.getDataRange().getValues();
    
    const uniqueStudents = {};
    for (let i = 1; i < thiData.length; i++) {
      let stId = String(thiData[i][2]).trim();
      let total = parseInt(thiData[i][18]);
      let result = String(thiData[i][19]).trim();
      let timeStr = String(thiData[i][1]).trim();
      if (stId) {
        if (!uniqueStudents[stId] || i > uniqueStudents[stId].rowIndex) {
          uniqueStudents[stId] = { total, result, rowIndex: i };
        }
      }
    }
    
    const courseMap = {};
    for (let i = 1; i < courseData.length; i++) {
      let cId = String(courseData[i][0]).trim();
      let cName = String(courseData[i][1]).trim();
      if (cId) courseMap[cId] = cName;
    }
    
    const studentCourseMap = {};
    for (let i = 1; i < memberData.length; i++) {
      let stId = String(memberData[i][0]).trim();
      let cId = String(memberData[i][5]).trim();
      if (stId && cId) studentCourseMap[stId] = cId;
    }
    
    const courseCounts = {};
    for (let cId in courseMap) { courseCounts[cId] = 0; }
    
    let resultCounts = { "Good": 0, "Fair": 0, "Poor": 0 };
    let totalUniqueStudents = 0;

    for (let stId in uniqueStudents) {
      totalUniqueStudents++;
      let cId = studentCourseMap[stId];
      if (cId && courseCounts[cId] !== undefined) {
        courseCounts[cId]++;
      }
      let resText = uniqueStudents[stId].result;
      if (resText.includes("Good") || resText.includes("มากกว่า")) resultCounts["Good"]++;
      else if (resText.includes("Fair") || resText.includes("เท่ากับ")) resultCounts["Fair"]++;
      else resultCounts["Poor"]++;
    }
    
    const summaryData = [];
    for (let cId in courseCounts) {
      summaryData.push({
        courseId: cId,
        courseName: courseMap[cId],
        count: courseCounts[cId]
      });
    }
    summaryData.sort((a, b) => b.count - a.count);
    
    return {
      success: true,
      totalAssessedUnique: totalUniqueStudents,
      summaryData: summaryData,
      resultCounts: resultCounts
    };
  } catch (error) {
    return { success: false, message: error.toString() };
  }
}
